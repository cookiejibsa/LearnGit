01

공동 작업자 추가하기

Collaborator 초대 — GitHub 저장소 Settings에서 진행
💡 핵심 원칙

GitHub의 공개(Public) 저장소는 누구나 소스를 볼 수 있고 clone도 가능합니다.
하지만 commit/push 권한은 승인된 공동 작업자에게만 있습니다.
무분별한 소스 수정을 막기 위해 반드시 협업자를 공식 초대해야 합니다.

1

저장소 설정 진입

팀장이 만든 저장소(예: projecttest)로 이동 후 상단 메뉴 Settings 클릭

2

Manage access 이동

좌측 메뉴에서 Collaborators (구 Manage access) 클릭 → Invite a collaborator 버튼

3

팀원 검색 및 초대

팀원의 GitHub username 또는 이메일 주소로 검색해서 초대 전송

4

팀원의 초대 수락

초대받은 팀원은 이메일 또는 GitHub 알림에서 Accept Invitation 클릭으로 수락
수락 완료 후 공동 작업자 목록에 표시됩니다.

⚠️ 주의

초대를 수락해야만 해당 저장소에 push 권한이 생깁니다. 수락하지 않은 상태에서는 push가 거부됩니다.

02

작업 환경 구성하기

각 팀원의 로컬 저장소 초기화 및 사용자 정보 설정

팀장과 모든 팀원이 각자의 컴퓨터에서 아래 명령어를 실행해 지역 저장소를 만들고 사용자 정보를 등록합니다.

Terminal — 모든 팀원 공통 실행 
# 1. 새 지역 저장소 생성 (프로젝트 폴더 자동 생성)
$ git init projecttest

# 2. 해당 폴더로 이동
$ cd projecttest

# 3. 이 저장소에서 사용할 이름 설정
$ git config user.name "사용자 이름"

# 4. 이 저장소에서 사용할 이메일 설정
$ git config user.email 이메일주소
💡 --global vs 로컬 설정

git config user.name은 해당 저장소에만 적용됩니다.
모든 저장소에 동일하게 적용하려면 git config --global user.name "이름"처럼 --global 플래그를 사용하세요.

03

원격 저장소에 첫 Commit & Push

팀장이 원격 저장소를 연결하고 최초 파일을 업로드
Terminal — 팀장 PC 
# 1. 원격 저장소(origin) 연결
$ git remote add origin https://github.com/팀장/projecttest.git

# 2. 파일 생성 (vim 에디터)
$ vim viewtest.txt

# 3. 스테이징 (모든 파일 추가)
$ git add .

# 4. 커밋 (메시지와 함께)
$ git commit -m "viewtest"

# 5. 원격 master 브랜치로 push (-u 옵션: 기본 upstream 설정)
$ git push -u origin master
파일 생성
→
git add .
→
git commit
→
git push
→
GitHub 업로드 완료 ✓
💡 -u 플래그란?

git push -u origin master의 -u는 upstream을 설정한다는 의미입니다.
이후에는 git push만 입력해도 자동으로 origin master로 push됩니다.

04

공동 작업자 PC에 원격 저장소 Clone

팀원들이 원격 저장소를 자신의 PC로 복제

협업을 시작하기 전, 팀원들은 각자의 PC에서 원격 저장소를 clone해야 합니다. 이 과정은 최초 1회만 수행합니다.

Terminal — 팀원 1, 팀원 2 PC (각각 실행) 
# 원격 저장소 전체를 현재 폴더에 복제
$ git clone https://github.com/팀장/projecttest.git

# clone하면 자동으로 'projecttest' 폴더가 생성됩니다
# 폴더 안에 원격 저장소의 모든 파일과 커밋 이력이 포함됩니다
✅ clone vs init 차이

git init: 빈 저장소를 새로 만들 때 사용
git clone: 이미 존재하는 원격 저장소를 내 PC로 복사할 때 사용
협업에서 팀원은 항상 clone을 사용합니다.

05

첫 번째 커밋이 아니라면 Pull 먼저

반드시 최신 상태를 받아온 후 작업 시작
🚨 협업의 핵심 규칙

여러 명이 동시에 push하면 충돌(Conflict)이 발생합니다.
작업 전 항상 git pull 먼저! — 이 습관 하나가 충돌을 90% 방지합니다.

📌 실제 충돌 시나리오

상황 1 — 팀장
viewtest.txt 수정 후 commit & push 완료
원격 저장소에는 "viewtest 수정" 커밋이 올라가 있음
상황 2 — 팀원 1 (오류 발생)
pull 없이 viewtest02.txt 만들고 push 시도
원격 저장소에 새 커밋이 있는데 내 PC에 없으므로 ! [rejected] 에러 발생
해결 — 팀원 1
git pull → vim 자동 실행 → commit → push 성공
pull로 최신 내용을 합친 후 다시 push
! [rejected] master → master (fetch first)
error: failed to push some refs to 'origin'
hint: Updates were rejected because the remote contains work that you do
hint: not have locally. Integrate the remote changes (e.g.
hint: 'git pull ...') before pushing again.
Terminal — 충돌 해결 흐름 (팀원 1) 
# 파일을 만들었으나 push 전에 pull을 먼저!
$ vim viewtest02.txt
$ git add viewtest02.txt
# → push 하면 [rejected] 에러 발생!

# 해결: pull로 원격의 최신 커밋을 당겨옴
$ git pull
# vim이 열리면 merge 커밋 메시지 작성 (기본 메시지 OK)

# 이제 나의 커밋을 올릴 수 있음
$ git commit -m "viewtest02"
$ git push -u origin master
  • 작업 시작 전 항상 git pull로 최신 상태 동기화
  • pull 시 vim이 자동 실행되면 merge commit 메시지 저장 후 종료 (:wq)
  • pull 후 충돌(conflict)이 있으면 파일 직접 편집 후 다시 add → commit
  • push 성공 후 다른 팀원들에게 알리는 것이 좋은 습관
06

새로운 브랜치로 Push하기

각 팀원이 독립 브랜치에서 작업하는 표준 협업 패턴
💡 왜 브랜치를 사용하나요?

협업에서는 각자 다른 기능(Feature)을 동시에 작업합니다.
모두 master에서 작업하면 충돌이 잦고 위험합니다.
각자의 브랜치에서 작업 → 완료 후 Pull Request로 master에 병합하는 것이 표준입니다.

Terminal — 브랜치 생성 및 Push 전체 흐름 
# 1. 최신 상태 동기화 (항상 먼저!)
$ git pull

# 2. 새 브랜치 생성 & 이동 (-b 플래그: 생성+이동 동시)
$ git checkout -b team1
# 이미 브랜치가 있다면: git checkout team1

# 3. 해당 브랜치에서 파일 작업
$ vim viewtest03.txt
$ git add viewtest03.txt
$ git commit -m "viewtest03"

# 4. 원격 저장소에 브랜치까지 함께 Push
$ git push origin team1
# origin(원격저장소)에 team1 브랜치를 새로 만들어 push
✅ Push 성공 후 확인

GitHub 저장소 웹 페이지에서 파일 목록 위 "2 branches"를 클릭하면
방금 push한 team1 브랜치가 목록에 나타납니다.

명령어 의미
git checkout -b team1 team1 브랜치 새로 생성 + 이동 (두 동작을 한 번에)
git checkout team1 이미 있는 team1 브랜치로 이동만 함
git branch 현재 존재하는 브랜치 목록 확인
git push origin team1 원격 저장소에 team1 브랜치를 업로드
07

Pull Request로 Branch Merge하기

코드 리뷰를 거쳐 master 브랜치에 병합

브랜치를 push했다고 master에 반영되는 것이 아닙니다. Pull Request(PR)를 통해 코드 리뷰 및 병합 요청을 보내야 합니다.

📤
New PR 생성
브랜치 옆 "New pull request" 클릭
💬
메시지 작성
변경 내용 설명 후 "Create pull request"
👀
리뷰 & 토론
팀원과 댓글로 피드백 주고받기
✅
Merge 완료
"Confirm merge"로 master에 병합
GitHub 웹 UI — Pull Request 단계별 진행 
1단계: 저장소 상단 "Pull requests" 탭 클릭
2단계: 등록된 PR 목록에서 해당 PR 클릭
3단계: 변경 내용 확인 후 "Merge pull request" 클릭
4단계: commit 메시지 확인/수정 후 "Confirm merge" 클릭
5단계: master 브랜치에 병합 완료!

확인방법:
  - "2 branches" 클릭 → team1 브랜치 상태가 'merged'로 표시
  - master 파일 목록에 viewtest03.txt가 나타남
✅ Merge 완료 후

병합된 브랜치는 merged 상태로 표시되며, 어떤 협업자가 병합했는지도 기록됩니다.
병합 후에는 불필요한 브랜치를 Delete branch로 삭제해 저장소를 깔끔하게 유지하세요.

전체 요약

GitHub 협업 전체 흐름

팀 프로젝트에서 반복되는 표준 사이클입니다.

STEP 1 — 최초 1회
팀장: 저장소 생성 → 팀원 초대 → 팀원: 초대 수락
STEP 2 — 최초 1회
모든 팀원: git clone으로 저장소 복제
STEP 3 — 매 작업마다 반복
git pull → 브랜치 생성/이동 → 작업 → add → commit → push
STEP 4 — 기능 완료 시
GitHub에서 Pull Request 생성 → 리뷰 → Merge
STEP 5 — Merge 후
다시 git pull로 master 최신화 → 다음 작업 반복
🔑 협업 황금 규칙 3가지

1. 작업 전 항상 git pull — 충돌을 미연에 방지
2. 기능별로 브랜치를 분리 — master 직접 수정 금지
3. Pull Request로 코드 리뷰 — 품질 유지와 지식 공유
